왜 OBSERVER인가?

Dexterous manipulation 연구는 체크포인트를 빠르게 쌓습니다 — reward shaping ablation, seed 민감도 실험, 커리큘럼 반복. 사람이 직접 영상을 보며 비교하는 방식에는 근본적인 한계가 있습니다. OBSERVER는 정량적·자동화된 평가로 그 한계를 제거합니다.

❌ 수동 리뷰
📹Isaac 뷰포트에서 30초 영상 시청
🤔"오늘은 좀 더 안정적으로 보이는데..."
🎲마지막에 본 영상이 가장 잘 기억됨 (최신 편향)
😴20번째 리뷰는 첫 번째만큼 예리하지 않음 (피로)
🚨slip, 관절 스파이크, 에너지 과다는 눈에 보이지 않음
✅ OBSERVER
📊8개 정량 지표 자동 집계
🔍6-class 규칙 체인으로 실패 유형 자동 분류
🗺️roll × pitch 히트맵으로 취약 포즈 구간 시각화
🏆success_rate + slip + energy + pose error 다목적 랭킹
📄차트·비디오·히트맵 포함 HTML 리포트 자동 생성

평가 병목 시나리오

🔬 Scenario A

Ablation Study

contact force weight × 4 / slip penalty × 3 / curriculum × 2 = 24개 설정. 체크포인트 1k·3k·5k·10k → 96개. 당신에게는 오후 하나.

🎲 Scenario B

Seed Sensitivity

10개 시드로 재학습. 10개 실행의 수렴 패턴을 수동 비교하는 것은 인지적으로 불가능합니다.

🔁 Scenario C

Reward Design Loop

보상 함수 조정 → 밤새 학습 → 영상 시청 → 불확실 → 반복. 1주일 후 30개+ 체크포인트. 어떤 보상이 효과가 있었고 왜인지?

🧪 Scenario D

Architecture Search

네트워크 depth × 3 / latent dim × 4 / tactile 유무. 질문은 "무엇이 실패했고, 포즈 공간의 어디에서" 입니다.

success_rate 뒤에 숨은 위험들

숨겨진 문제success_rate실제 결과
에피소드당 slip 3회표시 안 됨수 시간 만에 손끝 액추에이터 마모
roll ∈ [−20°, 20°]에서만 동작표시 안 됨암기, 일반화 아님 — 취약한 정책
특이점 근방 관절 속도 스파이크표시 안 됨실제 로봇 하드웨어 안전 위험
에너지 소비 baseline 대비 5배표시 안 됨배터리 방전, 발열 과부하
⚠️

핵심 인사이트: 30초 뷰포트 영상에서 좋아 보이는 정책이 실제 하드웨어에서는 위험할 수 있습니다. 이 문제를 잡는 유일한 방법은 다수 에피소드에 걸친 체계적이고 정량적인 분석입니다.

7단계 파이프라인

명령 한 줄. 모든 것이 자동화됩니다.

bash

    --auto_select --select_weights hardware_safe

컴포넌트 구조

각 컴포넌트는 독립적이며, PipelineOrchestrator가 체크포인트당 이 단계들을 조율합니다.

📦

MetricsCollector

에피소드 집계 지표 8개 수집. episodes.json에서 성공률·접촉력·slip 등을 계산해 metrics.json에 저장합니다.

🔍

FailureModeClassifier

우선순위 규칙 체인으로 에피소드별 실패 유형 분류. 학습 데이터 불필요, 체크포인트 0부터 즉시 동작합니다.

🗺️

StateCoverageAnalyzer

roll × pitch 초기 포즈 공간에서 정책이 어디서 실패하는지 히트맵으로 시각화합니다.

🎬

CameraController + VideoRecorder

Isaac Sim 5개 시점(front, side, top 등) + 2×3 그리드 비디오 저장. Replicator 기반 캡처.

📡

ExperimentTracker

TensorBoard 자동 감지 및 로깅. tensorboard가 없으면 조용히 건너뜁니다.

🏆

CheckpointSelector

다목적 가중치 점수로 top-k 체크포인트 선정. 프리셋: balanced / hardware_safe / performance_first.

📄

ReportGenerator

차트·파이 그래프·비디오·히트맵이 포함된 self-contained HTML 리포트. 브라우저 하나로 모든 결과를 봅니다.

🔄

PipelineOrchestrator

체크포인트 1개당 위 단계들을 순서대로 조율. 에러가 발생해도 다음 체크포인트로 계속 진행합니다.

통합 계약 (Integration Contract)

OBSERVER는 사용자의 policy를 직접 로드하지 않습니다. eval 스크립트와 record 스크립트를 서브프로세스로 실행하고, 두 스크립트가 정해진 계약(contract)을 만족하기만 하면 됩니다. 어떤 프레임워크(PPO, RSL-RL, CleanRL 등)든 동작합니다.

💡

계약의 핵심: eval 스크립트가 metrics.json과 episodes.json을 지정된 스키마로 출력하면 됩니다. 나머지는 OBSERVER가 처리합니다.

설정 파일

yaml
# observer/configs/eval_config.yaml
runtime:
  task:          "<your-task-id>"
  eval_module:   "your_pkg.scripts.eval"     # python -m 으로 실행
  record_script: "your_pkg/scripts/record.py"
  isaac_lab_path: "${ISAACLAB_PATH}/isaaclab.sh"

eval 스크립트가 출력해야 하는 것

파일필수 여부필수 필드
metrics.json✅ 필수checkpoint, num_episodes, success_rate
episodes.json권장length, success per episode — 실패 분류·히트맵에 필요

어댑터 예시 (sharpa-rl-lab)

실제 통합 예시는 docs/adapters/sharpa.md를 참고하세요. 새 프레임워크 어댑터 작성 가이드는 docs/21_ADAPTER_GUIDE.md에 있습니다.

ℹ️

policy 로딩 실패 시: OBSERVER는 policy를 직접 로드하지 않습니다. runtime.eval_module에 지정된 사용자 측 스크립트가 체크포인트·normalization 복원을 담당합니다.

히트맵 읽기

🔴 빨간 구역 = 다음 커리큘럼 타깃.

roll ∈ [30°, 60°]에서 실패가 집중된다면 → 다음 학습에서 해당 범위를 더 많이 샘플링하세요.

메트릭 & 실패 분류

6-class 실패 모드 (우선순위 순)

우선순위모드규칙하드웨어 의미
1early_drop에피소드 길이 < 50 스텝그립 초기화 실패
2singularity_hit최대 관절 속도 > 5 rad/s액추에이터 과부하 위험
3late_slipslip 이벤트 ≥ 3점진적 그립 저하
4contact_loss후반부 평균 접촉력 < 0.01 N손끝 접촉 상실
5repose_failure최종 포즈 오차 > 임계값안정적 그립에도 불구한 태스크 실패
6timeout최대 스텝 도달, 다른 규칙 없음정책이 너무 느리거나 stuck

실패 분포로 할 일 정하기

지배적 실패 모드권장 조치
late_slip 비중 높음보상 함수의 slip penalty 강화
early_drop 많음초기 그립 / 커리큘럼 설계 재검토
singularity_hit 많음관절 속도 제한 / 특이점 회피 보상 추가
contact_loss 많음접촉력 보상 강화
repose_failure 많음목표 포즈 오차 보상 조정
timeout 압도적커리큘럼 단순화 또는 탐색 개선
💡

분류기는 학습 데이터가 없습니다. 체크포인트 0부터 즉시 사용 가능하고, 분류 근거가 완전히 설명 가능합니다 — "rule 3 매칭으로 late_slip 분류됨."

체크포인트 랭킹

단일 success_rate로 체크포인트를 선택하면 하드웨어에 위험한 정책을 배포할 수 있습니다. OBSERVER는 여러 지표를 가중 합산해 랭킹합니다.

점수 공식

Score = wsr · SRnorm − wslip · slipnorm − wenergy · Enorm − wpos · pos_errnorm

각 지표는 평가 배치 내 min–max 범위로 정규화됩니다.

프리셋 선택

프리셋언제 사용특징
balanced평소 실험 비교모든 지표 균등 가중
hardware_safe실제 하드웨어 배포 직전slip + energy 패널티 큼
performance_firstablation 연구, 순수 성공률 중심success_rate 비중 최대화

랭킹 실행

bash
# hardware_safe 프리셋으로 랭킹 + top-2 배포
observer --checkpoint_dir runs/ \
    --auto_select --select_weights hardware_safe --deploy_top_k 2

# 결과
# eval_results/best/rank01__model_6000.pth  ← 1위
# eval_results/best/rank02__model_4000.pth  ← 2위
# eval_results/best/selection_meta.json      ← 점수 + 지표 상세

상태 커버리지 히트맵

StateCoverageAnalyzer는 roll × pitch 초기 포즈 공간을 격자로 나눠 정책이 어디서 취약한지 시각화합니다. episodes.json에 init_roll_deg / init_pitch_deg가 있어야 생성됩니다.

핵심 설계 원칙

OBSERVER의 비자명한 설계 결정들입니다. 이 원칙들이 존재하는 이유를 이해하면 통합 시 시행착오가 줄어듭니다.

🔲
서브프로세스 경계 — policy를 직접 로드하지 않음

OBSERVER는 eval 스크립트와 record 스크립트를 서브프로세스로 실행합니다. 크로스-프레임워크 호환성 문제를 완전히 회피하고, OBSERVER 코드에 프레임워크별 import가 없으며, 사용자 스택과의 버전 충돌이 없습니다.

🌐
프레임워크 무관 (Framework-Agnostic)

PPO / RSL-RL / CleanRL / 어떤 프레임워크든 동작합니다. 요구사항은 eval 스크립트가 metrics.json과 episodes.json을 지정된 스키마로 출력하는 것뿐입니다.

📋
규칙 기반 실패 분류 — 학습 데이터 없음

FailureModeClassifier는 학습 데이터를 사용하지 않습니다. 우선순위 규칙 체인으로 동작하며 체크포인트 0부터 즉시 사용 가능합니다. ML 기반 분류기의 부트스트랩 문제가 없고, 모든 결정이 완전히 설명 가능합니다.

🧩
선택적 의존성 패턴 (Optional Deps)

tensorboard와 opencv-python은 선택적입니다. 없으면 해당 기능이 조용히 건너뛰어집니다. 핵심 기능은 최소 의존성으로 동작합니다.

🔑
지표 키 폴백 — 필드 누락 시 계속 진행

metrics.json에 필드가 없으면 해당 분석을 건너뛰고 나머지를 계속 처리합니다. 필수 필드는 checkpoint, num_episodes, success_rate 셋뿐입니다.

정책 로딩 실패

OBSERVER는 policy를 직접 로드하지 않습니다. runtime.eval_module에 지정된 사용자 측 스크립트가 체크포인트·normalization 복원을 담당합니다. docs/20_INTEGRATION_CONTRACT.md §1 참조.

coverage / 실패 분류가 비어있음

대부분 episodes.json 생성 실패가 원인입니다. eval 스크립트가 해당 파일을 쓰는지, 스키마가 EpisodeStats와 맞는지 확인하세요. sharpa 사용자: docs/adapters/sharpa.md 트러블슈팅 섹션 참고.

success_rate가 0.0으로 나옴

체크포인트 로딩 또는 observation normalization 복원 실패입니다. eval 스크립트를 단독 실행해서 확인하세요:

bash
python -m <eval_module> --task=... --load_path=... --num_episodes=5 \
    --metrics_output=/tmp/m.json --episodes_output=/tmp/ep.json --headless
cat /tmp/m.json | python -m json.tool
